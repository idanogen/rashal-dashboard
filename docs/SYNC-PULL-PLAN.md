## 2. היעד

```
Make (תרחיש מתוזמן) --GET OData--> Priority Connect
        |
        +--POST upsert--> Supabase
```

- **תדירות:** כל 10 דקות בימים א'-ה' + פעם ביום שישי ושבת.
- **endpoint (אומת 24/06):**
  `https://p.priority-connect.online/odata/Priority/tabb4ce6.ini/shaal`
- **גישה:** Priority Connect פתוח מכל IP (לקח #12) → Make יעבוד, אין בעיית whitelist כמו אצל זאבי.
- **דלתא לפי watermark:** `$filter=UDATE ge {since}` — reuse של תבנית metalpress (תרחיש 4808186): ה-watermark נשמר ב-DB שלנו ומוחזר מ-endpoint שלנו, לא state ב-Make. עמיד לריצה שנכשלה (הריצה הבאה משלימה לבד).
- **upsert לפי מפתח טבעי:** ORDNAME להזמנות, מספר קריאה לקריאות שירות.
- **בונוס מיידי:** CUSTNAME מגיע "בחינם" ב-pull → customer_number מתמלא, החיפוש לפי מספר לקוח מתחיל לעבוד.

## 4. שלב 1 — הכנת ה-DB

1. **עמודות מפתח טבעי:**
   ```sql
   ALTER TABLE orders ADD COLUMN priority_order_id text;
   ALTER TABLE service_calls ADD COLUMN priority_call_id text;
   CREATE UNIQUE INDEX orders_priority_id_key ON orders (priority_order_id) WHERE priority_order_id IS NOT NULL;
   CREATE UNIQUE INDEX service_calls_priority_id_key ON service_calls (priority_call_id) WHERE priority_call_id IS NOT NULL;
   ```
   (unique חלקי — שורות היסטוריות בלי מפתח לא נחסמות.)
2. **חוזה בעלות על עמודות (קריטי!):** באפליקציה משתמשים משנים order_status, delivery_date, customer_reply_status ועוד. ה-upsert מפריוריטי:
   - ב-INSERT: כותב הכל, כולל סטטוס פתיחה.
   - ב-UPDATE: מעדכן **רק** שדות שפריוריטי הבעלים שלהם (פרטי לקוח, כתובת, customer_number). **לעולם לא דורס** order_status / service_call_status / delivery_date / שדות תיאום.
3. **endpoint צד-שרת** (Vercel api, service-role — כמו api/heyy-*.ts הקיימים):
   - `GET /api/priority-sync` → מחזיר `{since}` = max(watermark) פחות 5 דק', בפורמט OData מוכן.
   - `POST /api/priority-sync` → מקבל מערך רשומות מ-Make, עושה upsert עם חוזה הבעלות. מאובטח בטוקן.

## 6. שלב 3 — מעבר: גם וגם + ניקוי כפילויות (הוחלט 03/07)

עידן בחר להריץ את שניהם במקביל על אותן טבלאות, והמערכת אחראית לנקות כפילויות.
הסיכון המובנה: ה-webhook מכניס שורות **בלי** מפתח טבעי, ה-pull מכניס **עם** מפתח.
בלי טיפול, כל הזמנה תיכנס פעמיים. הפתרון: **מנגנון אימוץ ב-upsert**.

**לוגיקת האימוץ (בתוך `POST /api/priority-sync`):**
לכל רשומה שמגיעה מה-pull:
1. חפש לפי המפתח הטבעי (`priority_order_id`). נמצא → UPDATE לפי חוזה הבעלות. סיימנו.
2. לא נמצא → חפש שורה **בלי מפתח** שנוצרה ב-3 הימים האחרונים עם אותו (שם לקוח + טלפון). נמצאה → **אמץ אותה**: הדבק לה את המפתח + השלם שדות חסרים (customer_number!). ככה שורת ה-webhook הופכת לשורת ה-pull בלי כפילות.
3. לא נמצאה → INSERT רגיל.

מהצד השני, שורת webhook שנכנסת **אחרי** ששורת ה-pull כבר קיימת תיתפס על ידי
ה-trigger הקיים של הדה-דופ (חלון 5 דקות, אותו זיהוי עמום) ותסומן `duplicate_of`.
בין ה-trigger למנגנון האימוץ, שני הכיוונים מכוסים.

**ריצת ניקוי יומית (בטחון נוסף):** שאילתה מתוזמנת שמאתרת זוגות (עם מפתח / בלי מפתח)
של אותו לקוח באותו יום שחמקו, ומסמנת את חסר-המפתח כ-duplicate_of. דוח שבועי לעידן.

**סדר ההפעלה:**
1. הכנת DB + endpoint (שלב 1).
2. backfill חד-פעמי: משיכת 60 יום אחורה לפי CREATEDDATE, עם אותה לוגיקת אימוץ (רוב ההיסטוריה הקיימת "תאומץ" ותקבל מפתחות + customer_number רטרואקטיבית).
3. הפעלת התזמונים. ה-webhook נשאר דולק.
4. אחרי שבוע-שבועיים יציבים: דוח פערים (מה ה-pull תפס שה-webhook פספס). אז מחליטים אם ומתי לכבות את החוקה בפריוריטי (תיאום מול הגורם הרלוונטי אצל רשל) — או להשאיר לתמיד כגיבוי זמן-אמת.
